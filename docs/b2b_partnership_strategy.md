# 컴포즈커피 / 쿠팡이츠 B2B 파트너십 기술 전략

## 🔥 앱스토어 따로 등록 안 하는 방식 (런타임 브랜딩)

### 결론: 가능하고, 오히려 이게 더 현실적입니다

프랜차이즈가 여러 개 생겨도 **앱은 "알바급여정석" 1개만 유지**하고, 로그인 후 매장의 프랜차이즈 소속에 따라 **UI가 자동으로 바뀌는 방식**입니다.

### 두 방식 비교

| | **A. Flavor (앱스토어 별도)** | **B. 런타임 브랜딩 (앱 1개)** ✅ 추천 |
|---|---|---|
| 앱 스토어 등록 | 프랜차이즈마다 별도 등록 | **1개만 유지** |
| 앱 아이콘/이름 | 프랜차이즈별 커스텀 가능 | "알바급여정석" 고정 |
| 로그인 화면 | 전용 브랜딩 | 로그인 후 브랜딩 전환 |
| 업데이트 배포 | N개 앱 모두 따로 배포 | **1번만 배포** |
| 앱스토어 심사 | N번 심사 | **1번 심사** |
| Apple 개발자 계정 | 프랜차이즈별 필요할 수 있음 | **1개면 충분** |
| 프랜차이즈 10개 추가 시 | 10개 앱 등록+관리 😱 | **코드 변경 0** 🎉 |
| 관리 난이도 | 프랜차이즈 수에 비례해 증가 | **항상 일정** |

> [!TIP]
> **프랜차이즈가 여러 개 생겨도 관리 난이도가 전혀 증가하지 않는 건 B 방식뿐입니다.**
> 프랜차이즈 추가 = Firestore에 문서 1개 추가. 끝.

### 런타임 브랜딩 작동 방식

```
점주가 "알바급여정석" 설치 (기존 앱 그대로)
        │
        ▼
  로그인 (Google/Apple)
        │
        ▼
  _AuthGate에서 user 문서 조회
  users/{uid} → storeId 확인
        │
        ▼
  stores/{storeId} 문서 조회
  → franchiseId: "compose_coffee" 발견!
        │
        ▼
  franchises/compose_coffee 문서에서 브랜딩 로드
  → primaryColor, logo, appBarColor, 본사공지 등
        │
        ▼
  ┌──────────────────────────────┐
  │  컴포즈 브랜드 색상의 메인화면  │
  │  상단: ☕ 컴포즈커피 OO점     │
  │  본사 공지 탭 자동 활성화      │
  │  본사 교육 콘텐츠 표시         │
  └──────────────────────────────┘
```

### Firestore 구조 (프랜차이즈 추가 = 문서 1개)

  │
  ├── coupang_eats/

  │   ├── primaryColor: "#FF5722"
  │   └── ...

      ├── name: "파리바게뜨"
      ├── primaryColor: "#1B3A6B"
      └── ...

```

```

### 프랜차이즈 N개 추가 시 관리 비교

```
                   Flavor 방식              런타임 브랜딩 방식
                   ─────────              ────────────────
프랜차이즈 1개:    앱 2개 관리              앱 1개 + DB 문서 1개
프랜차이즈 5개:    앱 6개 관리 😰           앱 1개 + DB 문서 5개
프랜차이즈 20개:   앱 21개 관리 🤯          앱 1개 + DB 문서 20개
프랜차이즈 100개:  불가능 💀               앱 1개 + DB 문서 100개 ✅

업데이트 시:
  Flavor →  21개 앱 모두 빌드+심사+배포 (최소 2주)
  런타임 →  1개 앱만 배포 (1일)

긴급 버그 수정 시:
  Flavor →  21개 핫픽스 🔥🔥🔥
  런타임 →  1개 핫픽스 ✅
```

### 관리 난이도 매트릭스

| 작업 | Flavor (앱 별도) | 런타임 (앱 1개) |
|------|:---:|:---:|
| 신규 프랜차이즈 추가 | 빌드+스토어등록 (3~5일) | **DB 문서 추가 (5분)** |
| 앱 업데이트 배포 | N번 ×2플랫폼 | **1번 ×2플랫폼** |
| 노동법 개정 반영 | N개 앱 모두 배포 | **1번 배포** |
| 프랜차이즈 브랜드색 변경 | 재빌드+재배포 | **DB 수정 즉시 반영** |
| 인증서/프로비저닝 관리 | N개 | **1개** |
| 앱스토어 심사 대응 | N번 | **1번** |

### 유일한 제한: 앱 아이콘과 앱 이름

런타임 방식의 **유일한 한계**는 스마트폰 홈 화면의 앱 아이콘과 앱 이름을 바꿀 수 없다는 것입니다.

```
홈 화면에서:
  Flavor   →  ☕ 컴포즈HR    (전용 아이콘+이름)
  런타임   →  📊 알바급여정석  (공통 아이콘+이름)
```

> [!IMPORTANT]
> **이건 오히려 장점이 될 수 있습니다!**
> 
> "알바급여정석"이라는 브랜드 인지도가 쌓이면, 프랜차이즈 점주들이 
> **"우리도 알바급여정석 쓰는데"**라고 말하게 됩니다.
> 컴포즈 전용 앱으로 만들면 이 바이럴 효과가 사라집니다.
> 
> **제안 멘트**: "점주님이 앱을 열면 컴포즈 브랜드로 보이고, 
> 본사 공지도 바로 뜨지만, 앱 자체는 '알바급여정석'이라는 검증된 플랫폼입니다.
> 마치 카카오톡 안에서 회사 채널을 쓰는 것처럼요."

### 프랜차이즈가 "절대 우리 전용 앱으로 해달라"고 하면?

그때만 Flavor로 전환하면 됩니다. `BrandConfig` 구조는 이미 준비되어 있으니:

```
Phase 1: 런타임 브랜딩으로 시작 (리스크 제로)
Phase 2: 대형 프랜차이즈가 전용 앱 요구 시 → Flavor 빌드 추가 (1~2주)
```

**즉, 런타임 브랜딩과 Flavor는 양자택일이 아니라 단계적 확장입니다.**

## 🎯 프랜차이즈별 기능 특화 (Feature Flags)

### 결론: 완전히 가능합니다. "기능 스위치" 방식으로.

브랜딩(색상/로고)뿐 아니라 **각 프랜차이즈가 원하는 기능을 켜고 끄는 것**도 Firestore 문서 설정만으로 됩니다. 코드 배포 없이, DB 값만 바꾸면 즉시 반영.

### 구체적 예시: 프랜차이즈마다 다른 요구사항

```
컴포즈커피:  "본사 공지/교육 배포 + 신메뉴 교육 필수 이수"
쿠팡이츠:   "배달 건수 연동 + 라이더 출퇴근 관리"
파리바게뜨:  "위생점검 체크리스트 + 지점코드 관리" (이미 branchCode 있음!)
GS25:      "야간 수당 자동계산 강조 + 재고 유통기한 관리"
```

### Firestore의 `features` 필드로 해결

```
franchises/
  ├── compose_coffee/
  │   ├── name: "컴포즈커피"
  │   ├── primaryColor: "#8B4513"
  │   ├── logoUrl: "https://..."
  │   │
  │   └── features:                    ← 🔑 기능 스위치!
  │       ├── hqNotices: true          ← 본사 공지 활성화
  │       ├── hqEducation: true        ← 본사 교육 콘텐츠
  │       ├── menuTraining: true       ← 신메뉴 교육 기능
  │       ├── expirationTracker: true  ← 유통기한 관리
  │       ├── deliveryStats: false     ← 배달 통계 (불필요)

  │       ├── branchCodeRequired: false
  │       └── customTabs: ["공지", "교육", "신메뉴"]  ← 탭 구성도 커스텀
  │
  ├── coupang_eats/
  │   ├── name: "쿠팡이츠"
  │   └── features:
  │       ├── hqNotices: true

  │

      └── features:
          ├── hqNotices: true

          ├── deliveryStats: false
          ├── hygieneChecklist: true    ← 파바만 ON

          └── customTabs: ["공지", "위생", "유통기한"]
```

### 앱 코드에서 활용하는 패턴

```dart
// 1. 프랜차이즈 설정을 앱 전체에서 접근 가능하게 (InheritedWidget)
class FranchiseProvider extends InheritedWidget {
  final Map<String, dynamic>? features;
  final String? franchiseName;
  final Color? brandColor;
  final String? logoUrl;

  // 편의 메서드
  bool hasFeature(String key) => features?[key] == true;
  List<String> get customTabs => 
    List<String>.from(features?['customTabs'] ?? []);
}

// 2. 화면에서 조건부 렌더링 (기존 코드 최소 변경)
class NoticeEducationTabScreen extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    final franchise = FranchiseProvider.of(context);
    
    return DefaultTabController(
      length: _buildTabs(franchise).length,
      child: Scaffold(
        appBar: AppBar(
          bottom: TabBar(
            tabs: _buildTabs(franchise),
          ),
        ),
        body: TabBarView(
          children: _buildTabViews(franchise),
        ),
      ),
    );
  }

  List<Tab> _buildTabs(FranchiseProvider? f) {
    return [
      const Tab(text: '공지사항'),          // 항상 표시
      const Tab(text: '전달사항'),          // 항상 표시
      if (f?.hasFeature('expirationTracker') ?? true)
        const Tab(text: '유통기한'),        // 프랜차이즈 설정에 따라
      if (f?.hasFeature('hqEducation') ?? false)
        const Tab(text: '본사 교육'),       // 본사 교육 ON인 경우만
      if (f?.hasFeature('hygieneChecklist') ?? false)
        const Tab(text: '위생점검'),        // 파바만 표시
      if (f?.hasFeature('deliveryStats') ?? false)
        const Tab(text: '배달현황'),        // 쿠팡이츠만 표시
    ];
  }
}
```

### 특화 기능의 3가지 레벨

| 레벨 | 설명 | 구현 방법 | 예시 |
|------|------|----------|------|
| **Lv1. 표시/숨김** | 기존 기능을 보이거나 숨기기 | `features` 플래그 | 유통기한 탭 ON/OFF |
| **Lv2. 설정값 변경** | 같은 기능인데 기본값이 다름 | `defaults` 필드 | 컴포즈: 정산일 25일 고정 |
| **Lv3. 전용 기능 개발** | 해당 프랜차이즈만의 신규 기능 | 코드 추가 (1회) | 쿠팡이츠 배달 연동 |

```
franchises/compose_coffee/
  ├── features:          ← Lv1: 기능 ON/OFF
  │   ├── hqNotices: true
  │   └── menuTraining: true
  │
  ├── defaults:          ← Lv2: 기본값 커스텀
  │   ├── payDay: 25              ← 컴포즈는 25일 급여
  │   ├── payPeriodStart: 1       ← 1일~말일 정산
  │   ├── payPeriodEnd: 31
  │   ├── businessType: "food"
  │   └── gracePeriodMinutes: 10  ← 지각 허용 10분
  │
  └── customModules:     ← Lv3: 전용 기능 (필요 시 개발)
      └── menuTraining:
          ├── enabled: true
          └── quizRequired: true  ← 교육 후 퀴즈 필수
```

### 관리 시나리오

**시나리오 1: 컴포즈에서 "유통기한 관리 빼주세요" 요청**
```
→ Firestore에서 features.expirationTracker = false 로 변경
→ 컴포즈 점주 앱에서 유통기한 탭 즉시 사라짐
→ 코드 배포: 불필요 ✅
```

**시나리오 2: 쿠팡이츠에서 "배달 통계 기능 추가해주세요" 요청**
```
→ 배달 통계 화면 개발 (1회)
→ features.deliveryStats = true 인 프랜차이즈만 표시
→ 다른 프랜차이즈에 영향: 없음 ✅
→ 나중에 요기요도 배달 통계 원하면? DB에서 ON만 하면 됨 ✅
```

**시나리오 3: 파리바게뜨에서 "점포 등록 시 지점코드 필수로" 요청**
```
→ 이미 branchCode 필드 존재!
→ features.branchCodeRequired = true 설정
→ StoreInfoPage에서 해당 플래그 확인 → 필수 입력으로 전환
→ 코드 변경: if문 1줄 ✅
```

### 핵심 원칙

> [!TIP]
> **"공통 기능은 모두가 쓰고, 특화 기능은 스위치로 켠다"**
> 
> - 급여 계산, 출퇴근, 노무서류 → **공통** (모든 프랜차이즈 동일)
> - 본사 공지, 교육, 위생점검, 배달 통계 → **스위치** (프랜차이즈별 ON/OFF)
> - 새 기능 개발은 1번만 하고, 누가 쓸지는 DB에서 결정

## 📤 본사 교육/공지 콘텐츠 업로드: 누가, 어떻게?

### 3가지 옵션 비교

### 시장 규모: 알바급여정석이 노릴 수 있는 파이

```
한국 주요 프랜차이즈 매장 수 (2026년 기준, 실제 데이터):

편의점:
  CU             18,711개
  GS25           ~18,000개
  세븐일레븐       ~13,000개

커피:
  메가MGC커피     3,000개+
  컴포즈커피       3,184개    ← 우리의 1차 타겟
  이디야          2,581개
  빽다방          ~2,700개

외식:
  BBQ            ~1,800개

  쿠팡이츠 입점 매장  수만 개    ← 우리의 2차 타겟

```

### 경쟁사 가격 참고 (Anchor 등)

```
기존 프랜차이즈 교육 플랫폼 (예: emcast Anchor 3.0):

  - Standard: 월 수십만원~ (소규모 본사용)
  - Enterprise: 200개 매장 이상 맞춤 견적
  - 법정 의무교육 콘텐츠: 별도 유료
  - 슈퍼바이저 전용 앱: 별도 유료
  - 초기 도입비(Setup Fee): 별도

  → 공개 가격 없음 = "비싸서 못 보여줌" 🤑
  → 우리의 기회: 투명하고 합리적인 가격으로 차별화
```

> [!IMPORTANT]
> **우리의 결정적 차별점:**
> 
> 경쟁사(Anchor 등)는 **교육 관리만** 합니다.
> 알바급여정석은 **교육 + 급여 + 출퇴근 + 노무서류 + 근로계약**이 이미 통합되어 있습니다.
> 
> 즉, "교육 플랫폼"이 아니라 **"올인원 매장 노무관리 플랫폼"**입니다.
> 교육은 그 중 하나의 모듈일 뿐이므로, 가격 경쟁력이 압도적입니다.

### 컴포즈 본사에 제안할 가치(Value Proposition)

| 현재 컴포즈의 방식 (추정) | 알바급여정석 도입 시 |
|--------------------------|---------------------|
| 카카오톡/메일로 공지 전달 → 읽었는지 확인 불가 | 앱 내 공지 → **수신 확인 실시간 추적** |
| 교육은 점주 재량 → 미이수 파악 어려움 | 본사 교육 콘텐츠 → **이수율 자동 집계** |
| 법정 교육 관리 = 본사 인사팀 수작업 | **미이수 자동 경고 + 리포트** |
| 점포별 급여 현황 = 점주가 별도 보고 | **전 점포 인건비 실시간 대시보드** |

### 제안 시 킬러 한 마디

> "지금 카톡으로 보내시는 공지, 점주님이 읽었는지 알 수 있나요? 알바생까지 전달됐는지요?
> 저희 시스템을 쓰면 **본사에서 공지 올리면 전국 점포 앱에 바로 뜨고, 누가 읽었고 안 읽었는지 실시간으로 보입니다.**
> 교육도 마찬가지예요. 본사에서 올린 교육을 **누가 수료하고 안 했는지 전국 단위로 바로 집계**됩니다."
